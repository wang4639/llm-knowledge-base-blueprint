# 架构

## 核心思想

这套系统不是向量数据库，也不是“把所有文件交给模型全文扫描”。它是一条可追溯的知识编译流水线：

```text
只读原始资料
  → 结构化来源笔记
  → 跨来源 Wiki
  → 问答索引与问答卡
  → 文章、方案和决策输出
```

## 数据与控制分离

- 数据层：`10-来源笔记`、`20-Wiki`、`30-输出`；
- 控制层：`00-系统` 中的规则、模板、清单和配置契约；
- 观察层：`90-维护报告`；
- 外部原件：由 `config/knowledge-base.local.json` 指向，始终只读且不在仓库内。

## 两条运行回路

### 问答回路

`index → 板块 index → 问答索引 → 问答卡 → 少量来源`

它把高频问题编译成稳定卡片，降低每次问答的上下文开销，同时保留来源链接。

### 摄入回路

`检测差异 → 安全门禁 → 来源笔记 → Wiki → 问答卡影响检查 → 清单 → 动态验证`

新资料不会“自动等于新知识”。它必须说明对旧结论的关系，并保留冲突和不确定性。

## 为什么不默认使用 RAG

本项目优先解决信息结构、证据边界和维护纪律。只有当资料规模和查询形态证明 Markdown 索引不够时，才把 RAG 作为可替换扩展，而不是本版本依赖。
