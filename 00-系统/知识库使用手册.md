# LLM 知识库完整使用手册

本手册面向使用 Codex 或 Claude Code 的普通用户。你不需要先会写代码：安装后，把本项目目录作为 Agent 的工作目录，再用本手册中的提示词完成问答、资料摄入和维护即可。

这套系统是本地 Markdown 知识库，不是在线服务。Codex / Claude Code 负责阅读规则、整理资料和维护 Wiki；Obsidian 只负责更方便地浏览与编辑，属于可选工具。

## 1. 先理解四个入口

- [`../index.md`](../index.md)：知识库总首页，日常问答从这里开始；
- [`AI Agent 使用指南.md`](<AI Agent 使用指南.md>)：Agent 的问答路径与写入边界；
- [`../AI-REPRODUCE.md`](../AI-REPRODUCE.md)：安装者或部署 Agent 的完整复现协议；
- 本手册：给使用者看的安装后操作教程。

五层目录各自只做一件事：

```text
00-系统/       规则、模板、清单和 Agent 指令
10-来源笔记/   每份资料的结构化记录与证据入口
20-Wiki/       跨来源整理后的主题、概念、索引和问答卡
30-输出/       文章、方案、决策记录等可复用成品
90-维护报告/   检查结果、冲突和摄入报告
```

日常回答默认走：

```text
index.md
→ 20-Wiki/<板块>/index.md
→ 问答索引.md
→ 问题与假设/<问答卡>.md
→ 证据不足、存在冲突或需要溯源时，再读取 1–3 份来源笔记
```

## 2. 安装前准备

需要：

- Windows PowerShell 5.1 或 PowerShell 7；
- Git；
- 能访问本地文件的 Codex 或 Claude Code；
- 一个准备安装知识库的空目录；
- 一个或多个只放 Markdown 原件的资料目录。

可选：

- Obsidian，用项目根目录作为 Vault 打开；
- 支持定时任务的 Agent 客户端，用于每周维护与增量摄入。

仓库本身不需要 API Key。Codex 或 Claude Code 的登录、订阅和模型配置属于客户端自身配置，不应写入这个知识库。

## 3. 安装

### 方法 A：交给 Agent 自动安装

复制根目录 [`../README.md`](../README.md) 中“让 Codex / Claude Code 自动安装”的完整提示词，发送给 Agent。安装过程中只需要回答三个问题：

1. 安装到哪个空目录；
2. 哪些 Markdown 资料目录允许只读访问；
3. 需要哪些知识板块，例如“人工智能”“学习方法”“项目管理”。

如果暂时没有资料，可以让 Agent 在知识库目录之外创建一个空资料入口。不要把真实资料塞进公开蓝图仓库后再推回 GitHub。

### 方法 B：手动安装

克隆仓库并进入项目根目录：

```powershell
git clone https://github.com/wang4639/llm-knowledge-base-blueprint.git
Set-Location ".\llm-knowledge-base-blueprint"
```

复制本地配置：

```powershell
Copy-Item ".\config\knowledge-base.example.json" ".\config\knowledge-base.local.json"
```

编辑 `config/knowledge-base.local.json`：

```json
{
  "schema_version": 1,
  "max_batch_size": 30,
  "allowed_extensions": [".md"],
  "source_roots": [
    {
      "id": "inbox",
      "path": "..\\Knowledge-Inbox"
    }
  ],
  "boards": ["人工智能", "学习方法"],
  "source_note_prefix": "10-来源笔记"
}
```

配置规则：

- `id` 是资料入口的稳定代号，只能唯一；
- `path` 可以是绝对路径，也可以是相对项目根目录的路径，目录必须已经存在；
- `boards` 至少有一个板块，不能重名；
- `max_batch_size` 必须是 1–30；
- 公开提交中不要加入 `knowledge-base.local.json`，它已被 `.gitignore` 排除。

初始化并验证：

```powershell
& ".\scripts\Initialize-KnowledgeBase.ps1" -ProjectRoot "." -ConfigPath ".\config\knowledge-base.local.json"
& ".\scripts\Validate-KnowledgeBase.ps1" -ProjectRoot "." -ConfigPath ".\config\knowledge-base.local.json"
```

成功时会看到：

```text
INITIALIZATION_OK ...
VALIDATION_PASS ...
```

初始化器只创建缺失目录和初始文件；同名文件已存在时不会覆盖。

## 4. 第一次在 Codex / Claude Code 中打开

把项目根目录作为 Agent 的工作目录，然后发送：

```text
这是我的 LLM 知识库项目。请先完整阅读根目录 index.md、AI-REPRODUCE.md、
00-系统/AI Agent 使用指南.md、00-系统/知识库维护规则.md 和
00-系统/知识库使用手册.md。先只读检查，不要摄入、移动、删除、覆盖或改名任何文件。
请告诉我当前板块、资料入口、清单记录数、待录入差异，以及我可以怎样开始使用。
```

理想结果应包括：

- Agent 能说清五层目录职责；
- Agent 从 `index.md` 进入，而不是先扫描整个 `10-来源笔记`；
- Agent 报告五项资料差异；
- 没有得到授权前不做摄入或批量修改。

## 5. 日常提问

最简单的问法：

```text
请只依据这个知识库回答：<你的问题>。
先走 index → 对应板块 → 问答索引 → 问答卡；
只有证据不足、存在冲突或我要求溯源时，才读取最相关的 1–3 份来源笔记。
请区分知识库已有结论、来源观点和你的推断；材料不足就明确说不足。
```

需要证据时：

```text
请回答：<你的问题>。
给出直接结论、适用条件、例外、推荐行动，并链接最相关的来源笔记。
如果来源互相冲突，请并列冲突，不要替我静默裁决。
```

需要跨板块分析时：

```text
请从 <板块A> 和 <板块B> 综合回答：<你的问题>。
先分别定位两边的 Wiki/问答卡，再做交叉分析。
请标出哪些内容来自知识库，哪些是新的推断。
```

如果答案很有长期价值，可以继续说：

```text
请判断刚才的结论是否值得沉淀。
若值得，先提出准备更新的 Wiki、问答卡或 30-输出 文件清单和修改摘要；
未经我确认不要覆盖已有结论。确认后保留来源链接、条件、例外和冲突。
```

## 6. 添加新资料

### 第一步：放入资料入口

把 Markdown 原件放到 `knowledge-base.local.json` 中已授权的 `source_roots` 目录。原件可以在项目之外；系统通过根目录 ID、相对路径和 SHA256 记录它，不要求把原文复制进知识库。

### 第二步：只读检查

```powershell
& ".\scripts\Check-Inbox.ps1" -ProjectRoot "." -ConfigPath ".\config\knowledge-base.local.json"
```

结果写入 `00-系统/待录入资料检查.md`，包含五项门禁：

- `NewFiles`：全新资料；
- `DuplicateCopies`：内容哈希重复；
- `UpdatedFiles`：已登记路径的内容发生变化；
- `MissingFiles`：已登记资料不再存在；
- `SameNameCandidates`：同名但内容不同，需要人工判断。

只读维护发现任一差异时只能报告。只有 `NewFiles` 为 1–30 且其余四项全部为 0，专用摄入任务才可以继续。

### 第三步：让 Agent 摄入

```text
请执行一次受限增量摄入。
先完整阅读 AI-REPRODUCE.md 和 00-系统/自动增量摄入运行规则.md，
再运行 scripts/Check-Inbox.ps1。
只有新资料为 1–30 且重复、更新、缺失、同名候选全部为 0 时才继续；
否则只报告并停止。

对每份新资料：
1. 原件全程只读，处理前后核对 SHA256；
2. 按来源笔记模板区分事实、来源观点、AI 概括和 AI 推断；
3. “我的理解”和“我的评价”留空；
4. 优先更新已有 Wiki，判断对问答卡是无影响、强化、补充、冲突还是推翻；
5. 冲突并列保留，不覆盖旧结论；
6. 更新索引与日志后，用 Register-IngestedSource.ps1 追加清单；
7. 复查五项差异并运行 Validate-KnowledgeBase.ps1；
8. 在 90-维护报告 写一份不含本机绝对路径和秘密的结果。

开始前先告诉我本轮识别到的资料数量和目标板块。
```

一次摄入完成后，应满足：

- 原始资料哈希未变化；
- `10-来源笔记` 中出现对应结构化笔记；
- `20-Wiki` 的相关页面、问答卡和索引已更新；
- `00-系统/资料摄入总清单.md` 只追加了新行；
- 五项待录入差异重新归零；
- `Validate-KnowledgeBase.ps1` 输出 `VALIDATION_PASS`。

## 7. 什么时候写到哪里

| 内容 | 位置 |
|---|---|
| 单份资料的结构化记录、摘要、证据入口 | `10-来源笔记/` |
| 多份资料共同支持的概念、主题和稳定判断 | `20-Wiki/` |
| 高频问题的直接答案、判断标准、例外和行动 | `20-Wiki/<板块>/问题与假设/` |
| 准备交付或长期复用的文章、方案、决策记录 | `30-输出/` |
| 检查结果、失败、冲突和风险 | `90-维护报告/` |
| 使用者自己的理解和评价 | 由使用者亲自填写模板中的对应字段 |

`evergreen` 表示使用者确认过的长期有效内容，Agent 不得自行设置。

## 8. 两项自动化怎么用

项目提供两个调度器无关的 Prompt：

- `automations/weekly-maintenance.prompt.md`：每周只读检查，只报告，不摄入；
- `automations/weekly-ingestion.prompt.md`：门禁通过时最多摄入 30 份。

推荐把维护任务安排在摄入任务之前。两项任务使用同一个项目根目录，但权限必须分离。

你可以对支持定时任务的 Agent 说：

```text
请根据 automations/README.md 和其中两份 Prompt，为这个知识库配置两个每周任务。
先向我确认执行时间。维护任务必须只读；摄入任务只有门禁通过才可写入。
不要把本机绝对路径、模型名或任务 ID 写回仓库公开文件。
创建后告诉我任务名称、时间、工作目录和失败时如何通知。
```

若客户端不支持定时任务，手动执行两份 Prompt 也能得到相同的权限边界。

## 9. 每周维护

手动维护提示词：

```text
请执行本知识库的每周只读维护。
阅读 00-系统/知识库维护规则.md 和 automations/weekly-maintenance.prompt.md；
检查资料差异、索引覆盖、失效链接、孤立页面、重复来源、缺少证据的强判断和冲突结论。
只在 90-维护报告 中写脱敏报告，不摄入、不修复、不删除、不移动、不覆盖。
最后按严重程度给我列出需要人工决定的事项。
```

维护报告提出问题不等于自动修复授权。删除、移动、改名、合并、覆盖、处理更新资料和裁决冲突都必须单独确认。

## 10. 验证与排错

完整自检：

```powershell
& ".\scripts\Validate-KnowledgeBase.ps1" -ProjectRoot "." -ConfigPath ".\config\knowledge-base.local.json"
& ".\tests\Test-KnowledgeBase.ps1" -ProjectRoot "."
& ".\tests\Test-PublicRelease.ps1" -ProjectRoot "."
```

常见问题：

| 现象 | 原因与处理 |
|---|---|
| `本地配置不存在` | 复制 example 配置为 `knowledge-base.local.json`，再填写真实资料目录 |
| `授权资料目录不存在` | 检查路径拼写并先创建目录；不要让 Agent 猜路径 |
| `boards 至少需要一个板块` | 在本地配置中添加至少一个板块 |
| `动态验证失败：资料差异尚未归零` | 先查看 `待录入资料检查.md`，按差异类型处理，不要绕过门禁 |
| 出现重复、更新、缺失或同名候选 | 只报告，交给使用者决定；不要自动注册或覆盖 |
| PowerShell 脚本执行受限 | 可用 `powershell.exe -ExecutionPolicy Bypass -File <脚本>` 临时运行单个可信脚本，不必修改系统全局策略 |
| Agent 一开始扫描所有来源笔记 | 让它重新阅读 `AI Agent 使用指南.md`，强制按 Wiki-first 路由重答 |
| 回答像模型常识而非知识库内容 | 要求它标出知识库证据与 AI 推断；本地材料不足时必须明确说不足 |

产品测试使用纯虚构临时资料，不会读取或修改你的真实资料目录。

## 11. 隐私、Git 与备份

- 原始资料目录只读，不把原件全文复制到公开仓库；
- `knowledge-base.local.json`、`.env`、私钥和数据库文件不得提交；
- `10-来源笔记`、`20-Wiki` 和 `30-输出` 会包含你的知识内容；若你使用 Git 备份，远端必须由你自行确认是私有且访问权限正确；
- 推送前运行 `Test-PublicRelease.ps1` 只能做模板级规则扫描，不能替代你对私人内容的人工判断；
- 备份时应同时保留知识库目录和项目外的原始资料目录，并保持原路径或同步更新本地配置；
- 升级蓝图时先在新的空目录拉取新版并比较脚本与规则，不要用新版目录直接覆盖已有知识库。

## 12. 安全边界速查

Agent 可以自动做：

- 读取已授权资料；
- 创建缺失的结构；
- 在门禁通过时建立来源笔记、更新 Wiki、追加清单；
- 运行验证并写脱敏报告。

Agent 必须先征得你确认：

- 删除、移动、改名、合并或覆盖文件；
- 处理已更新、缺失、同名或冲突资料；
- 把内容标为 `evergreen`；
- 把你的知识内容推送到任何远端；
- 修改客户端全局配置、凭据或权限。

使用者负责：

- 决定资料目录与板块；
- 填写“我的理解”和“我的评价”；
- 裁决冲突与高风险变更；
- 确认备份、远端可见性和最终发布内容。
